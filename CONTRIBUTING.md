# Contributing to GameStudio

GS is a provider-neutral coordination framework for indie game development in
supported coding-agent harnesses. The canonical source is `.agents/`; `.claude/`,
`.codex/`, `.cursor/`, and future harness folders are adapters that should map
back to that source.
Contributions are welcome — bug fixes, new skills that fill a real gap, agent
improvements, and hook fixes. PRs that don't fit the framework's direction will
be closed without lengthy explanation.

## What Makes a Good PR

- **Bug fixes** — something is broken, here's the fix
- **New skills** that address a workflow gap not already covered
- **Improvements** to existing agents, skills, or hooks
- **Documentation corrections** — wrong info, broken references, outdated steps

Feature requests submitted as PRs will be closed. Open an issue instead.

**What this repo isn't:**
GS is the system that helps you build games, not a place to store the games
you build with it. GDDs, ADRs, PRDs, game concepts, level designs, narrative
docs, or any other output generated by GS for your own project won't be
merged here — keep those in your own repo.

## Provider-Neutral Contribution Model

Contributions should improve the studio workflow once, then expose it through
adapters where needed:

- Put canonical skills in `.agents/skills/<name>/SKILL.md`.
- Put canonical role definitions in `.agents/agents/`.
- Put canonical lifecycle scripts in `.agents/hooks/` and register them in
  `.agents/hooks.json`.
- Put canonical rules and templates in `.agents/rules/` and
  `.agents/docs/templates/`.
- Keep `.claude/`, `.codex/`, `.cursor/`, and similar folders as adapter layers.

Do not make a skill, agent, rule, or template behave differently because the
active model is Claude, GPT, Gemini, DeepSeek, GLM/Z.ai, Qwen, or local. Model
choice belongs in the harness or an optional gateway.

## The Non-Negotiable Technical Rules

These are the things that will get your PR rejected if you miss them.

**Skill files**
- Canonical skills live in `.agents/skills/<name>/SKILL.md`; adapter copies may
  live under `.claude/skills/` or another harness-specific directory. The
  subdirectory format is required.
- SKILL.md must include YAML frontmatter: `name`, `description`,
  `argument-hint`, and `allowed-tools`
- Do not hardcode provider or model names in canonical skills. If an adapter
  needs a reasoning tier hint, keep it advisory and let the active harness or
  gateway choose the actual model.

**Hooks**
- Canonical hooks live in `.agents/hooks/`; adapter hook copies must preserve
  the same behavior unless the difference is explicitly documented.
- Use `grep -E` — never `grep -P` (Perl regex breaks on Windows Git Bash)
- Include fallbacks for systems without `jq` or `python` installed
- Hooks run on every session start — they must exit quickly and gracefully
  (`exit 0`) when not applicable
- If you change hooks, harness adapters, gateway examples, or provider-neutral
  docs, run `python .agents/scripts/validate-compatibility.py`

**Agents**
- New agents must include a **Collaboration Protocol** section that describes
  how the agent asks questions and defers decisions to the user
- Agents must not modify files outside their documented domain without explicit
  user delegation

**Reference docs**
- If your PR adds or changes a skill, agent, or hook, update the matching
  provider-neutral reference doc in `.agents/docs/` (`agent-roster`,
  `skills-reference`, `hooks-reference`, or `rules-reference`). PRs that add
  things without updating the index will be
  sent back.

**Repository metadata**
- If your PR changes security-sensitive behavior, update `SECURITY.md` when the
  reporting scope, risk model, or contributor guidance changes.
- If your PR changes collaboration or review expectations, update this file,
  `.github/PULL_REQUEST_TEMPLATE.md`, and any relevant issue template.
- If your PR adds a new high-impact path, update `.github/CODEOWNERS`.

## The Collaborative Principle

GS is not an autonomous system. Every workflow follows:
**Question → Options → Decision → Draft → Approval → Write**

Skills and agents must ask before acting. Nothing writes to files without
explicit user confirmation. If your contribution has an agent making decisions
or writing files unilaterally, it won't be merged.

## Testing Your Changes

Run it in at least one supported coding-agent harness and confirm it works
end-to-end. For skills, invoke the skill and verify the output matches what the
skill claims to do. For hooks, trigger the relevant event and confirm the hook
fires correctly and exits cleanly.

For provider-neutral compatibility changes, run:

```bash
python .agents/scripts/validate-compatibility.py
```

Include a brief note in your PR description describing what you tested and
what the output looked like.

For README count changes, verify the canonical tree directly:

```powershell
(Get-ChildItem .agents/agents -File).Count
(Get-ChildItem .agents/skills -Directory).Count
(Get-ChildItem .agents/hooks -Filter *.sh -File).Count
(Get-ChildItem .agents/rules -Filter *.md -File).Count
(Get-ChildItem .agents/docs/templates -Filter *.md -File).Count
(Get-ChildItem .agents/docs/templates/collaborative-protocols -Filter *.md -File).Count
```

## Commit Format

Use [Conventional Commits](https://www.conventionalcommits.org/):

```
feat: add /retrospective skill for end-of-sprint reviews
fix: correct grep -P usage in session-start hook
docs: update skills-reference with new /qa-plan entry
```

Types: `feat`, `fix`, `docs`, `chore`, `refactor`, `test`

## PR Process

- Your PR will be auto-assigned to the maintainer via CODEOWNERS
- Reviews happen when they happen — this is a solo-maintained project
- If your PR sits open without feedback for a few weeks, a nudge comment is fine
- Merged contributors are credited in release notes

## Platform Compatibility

GS must work on Windows (Git Bash), macOS, and Linux. If your hook or
script uses anything platform-specific, it will be rejected. When in doubt,
test on Windows.
